01

The empty quadrant is real

The hypothesis was: nobody occupies GUI + text-native-project-file + CLI at once. The research supports it — the closest neighbors each miss exactly one axis.

openDAW is the sharpest data point Verified: an actively developed open-source web DAW (TypeScript ~76%, Rust ~21%, ~1.9k stars) that already ships a headless SDK — so a browser-audio engine with programmatic access is proven. But it stores projects as zipped bundles, so despite being open and headless-capable it deliberately does not sit in the text-native quadrant. That is the gap, occupied by no one.

Prior art for the pieces exists and is worth stealing from Single-source: DAWproject (Bitwig, MIT-licensed, XML schema, shipping in Studio One / Cubase / Bitwig) is the reference schema for representing real sessions in text; alsdiff and Ableton's own maxdiff prove "musical diff of a DAW project under git" is a real workflow — and alsdiff's key lesson transfers directly: match tracks/clips/devices by stable IDs, not position, so a rename or reorder doesn't explode the diff.

Verdict

Build the format. The quadrant is empty for structural reasons (everyone treats text as an export, never the working file), not because it was tried and failed. Borrow DAWproject's schema vocabulary; borrow alsdiff's ID-based matching.

02

The headless render path is already de-risked

The riskiest CLI assumption — "render a project to a WAV without a browser window" — turns out to have two independent working paths.

  • node-web-audio-api (IRCAM) — Rust implementation of Web Audio with a Node polyfill that installs the globals Tone.js expects. Its repo ships a working examples/tone.js running FM synths, Tone.Loop, and Transport in Node, and implements OfflineAudioContext — i.e. faster-than-real-time render to a buffer, straight to WAV. Single-source
  • Tone.js OfflineContext — Tone wraps OfflineAudioContext natively; .render() returns a buffer. A whole project renders programmatically. Single-source
  • Headless Chromium — the fallback, and the one I've already proven in this very project: BeatLab's smoke suite boots the real app in headless Chrome and drives the audio engine. If the Node bindings ever diverge from browser behavior, this path renders bit-identically to what users hear.
Verdict

beat render project.beat -o mix.wav --stems is a weeks-not-months task. Start on headless Chromium (guaranteed fidelity), migrate hot paths to node-web-audio-api for speed once it matters. One caveat surfaced Risk: standardized-audio-context, a Tone dependency, doesn't run in plain Node — which is exactly why the polyfill/Chromium route exists.

03

Where the web stack's ceiling actually sits

This is the "not a toy" question. The evidence draws a clean line: the browser is fine for a MIDI-and-synth DAW (which is what BeatLab already is) and hits real walls on recorded-audio production (warping, low-latency tracking, plugin hosting).

SubsystemPure web (AudioWorklet + WASM)Fallback path
Synth / MIDI / sequencingships today— openDAW: 27 devices, no WASM core yet
Stock FX (EQ/comp/reverb)yes — WASM DSPC/C++/Faust → WASM via Emscripten
Third-party pluginsWAM2 onlyTauri shell → native CLAP/VST3
Audio recording latency~30ms floornative audio (~10ms) needs Tauri
Latency compensationAPIs don't expose itnative driver reports true latency
Warping / time-stretchWASM libs, unprovenRubber Band / signalsmith in WASM
Multicore audiomulti-graph + SABring buffers over SharedArrayBuffer

The load-bearing numbers Single-source, W3C workshop: browser DAWs floor out around 30 ms round-trip latency where native needs ~10 ms, and recording-latency compensation is structurally blocked because MediaStreamSourceNode latency "is not exposed anywhere." The WAM-studio authors, who shipped a full browser DAW, name the same two culprits — the sandbox and latency compensation — as their hardest problems, explicitly not raw DSP power. And openDAW's own roadmap is the tell: it shipped a multitrack DAW with 27 devices, but audio fades, tempo-map automation, and pitch/stretch are all still 2026 roadmap items. Those are precisely the recorded-audio features, and they're last for a reason.

Architecture call

Ship pure-web for the MIDI/synth DAW; plan a Tauri shell as the "pro audio" tier. Tauri gives native-latency recording, real plugin hosting, and native time-stretch behind the same web UI and the same text file. The document format and CLI are the constant; the audio backend is swappable. This is also the honest answer to "not a toy": the toy/serious line is drawn by the backend, not the file format.

04

The AI-listening reality check — the finding that changes the design

You asked to make AI listening a core component. The research is unusually consistent and unusually sobering, and it confirms the instinct from our earlier "can you hear it?" conversation: today's audio-LLMs are bad ears. Building on them naively would make the feature a liability.

What the benchmarks say Single-source ×5, converging

  • Best models score ~52% on expert audio understanding (MMAU) vs 82% for humans; music is their single weakest domain, below speech and environmental sound.
  • The dominant error is perceptual, not reasoning: 55–64% of errors are the model mis-hearing the audio.
  • Several open models (MU-LLaMA, SALMONN) barely change their answer when the audio is replaced with noise — they're answering from text priors, not listening.
  • Fine-grained, time-localized judgments — exactly "the lead's attack is masking the kick" — are the worst-performing category (beat/melody/temporal tasks near the floor).

So an autonomous "render → let an audio-LLM critique → apply" loop, trusted blindly, would confidently hallucinate mix problems that aren't there. That's the trap. But the research also hands over the escape route — and it's the same shape as BeatLab's own lesson validators.

The architecture that works: measure first, model second

The credible path splits the job. Objective DSP metrics are the ground truth; the language model is a translator and advisor on top, never the sole judge:

  • Guardrail layer (deterministic DSP): LUFS / true-peak / crest factor (EBU R128), per-band spectral balance vs a reference track, a masking metric across stems, stereo width. These are cheap, exact, and don't hallucinate. This layer alone already powers commercial assistants.
  • Action layer (learned, interpretable): the auto-mix literature split matters here — prefer systems that predict effect parameters, not black-box audio. Diff-MST (ISMIR 2024) and the Apache-licensed automix-toolkit's Differentiable Mixing Console do exactly this: infer a mix from a reference and output human-adjustable EQ/comp/gain settings — which map straight onto your text project fields. Single-source
  • Language layer (LLM): turns metric deltas into readable suggestions and diffs ("bass is +4 dB in the 200–400 Hz band vs the reference → cut here"). The number is real; the model just narrates and proposes the edit.
Verdict

The AI-listening feature is buildable and genuinely differentiating — because your project is text and your renders are scriptable, every suggestion can land as a reviewable diff. But architect it metrics-first: DSP measurements are the source of truth and the guardrail; the audio-LLM is the narrator, not the ear. This is the BeatLab validator idea grown up — "check my mix against a target" instead of "check my work against a lesson."

05

Tiered build plan

Sequenced so each tier ships something whole, and the two hard commitments (native audio, learned auto-mix) come after the format and CLI have proven themselves.

MVPthe empty quadrant, occupied — pure web
  • The .beat text format — ID-matched, one-note-per-line, the GUI's canonical save target (lift BeatLab's store onto disk).
  • Local daemon — owns the file, serves the UI, two-way sync so vim edits hot-reload and knob-turns show in git diff.
  • CLI — render (headless Chromium → WAV/stems), inspect, set, and a musical diff.
  • Everything BeatLab already is: synths, sampler, piano roll, mixer, FX chains.
V1agent-native + the listening loop
  • MCP server over the CLI ops — Claude edits the file, renders, reads metrics, iterates.
  • Metrics engine — LUFS/true-peak/crest/spectral-balance/masking as a deterministic analysis pass (the guardrail layer).
  • AI critique — metric-grounded suggestions delivered as project diffs; opt-in "lint rules for mixes."
  • Real arrangement timeline (arbitrary length, tempo/time-sig changes) — the biggest pure-engineering lift.
Parity push"not a toy" — Tauri native tier
  • Tauri shell — native-latency recording, latency compensation, CLAP/VST3 hosting.
  • Warping / time-stretch via Rubber Band or signalsmith (WASM in web, native in Tauri).
  • Comping, audio-region editing, freeze/flatten/bounce with defined signal-path semantics.
  • Learned auto-mix (Diff-MST / DMC) producing parameter suggestions, not black-box audio.
  • Modulation system (Bitwig-style), macro racks, MPE, note probability.
Out of scopesay no on purpose
  • In-browser VST/AU hosting — will never work; that's what the Tauri tier is for. Web plugins = WAM2 only.
  • Beating Ableton at recorded-audio workflows — the positioning is "git-native groove/production tool for people who code + produce with AI," not a Pro Tools replacement.
  • Trusting an audio-LLM as an autonomous mix judge — the benchmarks forbid it; metrics stay in the loop.
  • A generator-code layer — you chose document-only for v1; the two-layer model stays on the shelf until the document format is proven.
06

Top risks, ranked

  1. Audio-LLM hallucination on mix judgments high

    The best-evidenced risk in the whole report. Mitigation is architectural, not incidental: DSP metrics are ground truth, the model narrates. Never let the LLM be the sole judge of a change.

  2. Recording latency & compensation on the web high

    Structurally unsolved in-browser (~30ms floor, latency not exposed by APIs). Only mitigation is the Tauri native tier — which means "serious recording" is gated behind a second runtime from day one of planning.

  3. Format churn as features deepen high

    A text format that's beautiful at MVP can calcify against timelines, warping, automation curves. Mitigation: version the schema from commit one, study how DAWproject/tracktionedit represent the full surface before locking v1.

  4. GUI↔file round-trip fidelity med

    Every GUI edit must serialize losslessly and every hand-edit must load — or trust in the format collapses. Mitigation: property-based round-trip tests (serialize→parse→serialize is identity) in CI from the start.

  5. Scope: "Ableton parity" is a decade for a team med

    The feature surface is enormous. Mitigation is the positioning itself — win the niche (code + AI-native) rather than the feature checklist. Parity is a direction, not a milestone.

  6. Node/browser audio divergence med

    node-web-audio-api may not render bit-identically to the browser. Mitigation: keep headless Chromium as the fidelity-guaranteed render path; treat Node bindings as a speed optimization to validate against it.

  7. Learned auto-mix below pro quality med

    Even the best deep-learning mixers still trail human engineers. Mitigation: ship it as suggestions the user accepts/rejects as diffs, never as a silent auto-apply.
